# Vibe Coding 自动化测试

## 定位

帮助 Vibe Coding 项目实现"测试 → 修复 → 再测试"的自动闭环：

1. 根据产品方案自动生成测试用例
2. 自动执行 Playwright 测试
3. 输出测试报告
4. AI 根据报告自动修复代码
5. 循环直到测试通过
6. 通知用户人工介入做最终检查

## 必要输入材料

启动测试前，按以下规则判断需要什么：

| 情况 | 处理方式 |
|------|------|
| 只有 URL，无产品方案 | ✅ 允许：先用 Playwright 探索页面主要流程，再生成用例 |
| 有产品方案 + URL | ✅ 最佳：直接基于方案生成用例 |
| 只有产品方案，无 URL/文件 | ❌ 缺页面：告知用户需要提供可访问的页面地址或 HTML 文件 |
| 什么都没有 | ❌ 告知用户至少需要一个可访问的页面地址 |

### 仅有 URL 时的探索策略

先花 3-5 步快速探索页面：
1. 导航到 URL，截图看整体结构
2. 找到主要入口按钮，点进去看核心流程
3. 走一遍完整主流程（如注册→创建→操作→提交）
4. 记录页面结构、主要功能模块、关键交互
5. 基于探索结果生成用例，展示给用户确认

## 流程概览

```
用户提供：产品方案 + HTML文件/URL
        ↓
Step 1：理解需求，生成核心测试用例（含 UI 自检用例）
        ↓
Step 2：对话确认，按反馈修改用例
        ↓
Step 3：用户确认后，Playwright 自动执行 + 逐页 UI 截图分析
        ↓
Step 4：输出测试报告（功能 + UI 合并）
        ↓
Step 5：有失败项？AI 自动修复代码 → 重新跑测试 → 循环
        ↓
Step 6：全部通过 → 通知用户人工介入做最终确认
```

## Step 1：生成测试用例

### 从自然语言提取测试维度

收到产品描述后，按以下维度拆解用例：

1. **核心功能**：主流程是否跑通（最高优先级）
2. **边界输入**：空值、超长、特殊字符、非法格式
3. **错误处理**：错误提示是否正确显示
4. **状态变化**：操作后 UI 状态是否正确更新
5. **交互细节**：按钮禁用/启用、加载状态、跳转逻辑

### 用例输出格式

每条用例包含：
- **编号**：TC-001、TC-002...
- **功能模块**：所属功能区域
- **测试描述**：一句话说明测试什么
- **操作步骤**：具体操作序列
- **预期结果**：明确的判断标准（可量化、可验证）
- **优先级**：P0 / P1 / P2

示例：
```
TC-001 | 登录 | 正确账号密码登录成功
步骤：输入正确账号 → 输入正确密码 → 点击登录
预期：跳转到首页，URL 变为 /home 或页面出现「欢迎」文字
优先级：P0
```

### 用例数量原则

- P0（核心主流程）：3-8 条，必须全部通过才算合格
- P1（重要功能）：按模块各 2-5 条
- P2（细节）：可选，不影响主报告结论

### UI 自检用例（固定追加，无需用户提供）

每次测试**自动追加**以下 UI 类用例，无需用户描述：

| 编号 | 描述 | 检查方式 |
|------|------|------|
| UI-001 | 各页面无白屏/空白异常 | 截图后 AI 视觉分析 |
| UI-002 | 文字无截断/溢出容器 | 截图后 AI 视觉分析 |
| UI-003 | 无元素重叠遮挡 | 截图后 AI 视觉分析 |
| UI-004 | 字体/文字正常渲染（无方块乱码） | 截图后 AI 视觉分析 |
| UI-005 | 按钮/图标显示正常 | 截图后 AI 视觉分析 |
| UI-006 | 颜色对比度无明显可读性问题 | 截图后 AI 视觉分析 |
| UI-007 | 整体布局无明显错乱 | 截图后 AI 视觉分析 |

UI 自检在功能用例执行**完成后**统一进行，对每个主要页面截图，用 `image` 工具做 AI 视觉分析。

## Step 2：对话确认

生成用例后，**以列表形式展示给用户**，说明：
- 覆盖了哪些功能模块
- P0 用例有几条
- 是否有理解不确定的地方（主动标出，请用户澄清）

等用户说「没问题」或「开始测试」后，进入 Step 3。

用户可能的反馈处理：
- 「少了 XX 功能」→ 补充对应用例
- 「这条不对」→ 修改该条用例
- 「去掉 XX」→ 删除对应用例
- 「加一条」→ 按用户描述新增

## Step 3：Playwright 自动执行

### 环境初始化

每次执行前先运行 browser-operation skill 的 setup.js 确保 Playwright MCP 就绪：

```bash
node <browser-operation-skill路径>/setup.js
```

### 判断页面类型，确定访问方式

收到用户输入后，先判断页面类型：

**类型 A：本地 HTML 文件（纯前端，无后台）**
- 用户提供的是文件路径（如 `/path/to/your/app.html`）或说「本地文件」
- 用 exec 起一个临时静态服务器：
```bash
npx serve <文件所在目录> --port 8788 --no-clipboard &
```
- 访问地址：`http://localhost:8788/<文件名>`
- 测试完成后不需要手动停止（进程随 session 结束）

**类型 B：已运行的 URL（有后台）**
- 用户直接提供 http/https 地址
- 直接 `browser_navigate url=<URL>`，无需额外处理
- 内网地址需提前确保测试环境网络可达

### 执行策略

1. **导航到页面**
```
browser_navigate url=<页面URL>
browser_wait_for time=2
browser_snapshot   # 确认页面加载完成
```

2. **逐条执行用例**
- 按 P0 → P1 → P2 顺序执行
- 每条用例执行后 snapshot 验证结果
- 记录：通过 ✅ / 失败 ❌ / 阻塞 ⚠️（依赖项失败导致无法执行）

3. **失败处理**
- 截图保存到 `screenshots/tc-xxx-fail.png`
- 记录失败原因（元素不存在 / 文字不符 / 跳转未发生）
- 继续执行下一条，不中断整体

4. **页面状态重置**
每条用例执行完后，如需回到初始状态：
```
browser_navigate url=<页面URL>   # 重新加载
browser_wait_for time=1
```

### 页面状态重置

每条用例执行完后，如需回到初始状态：
```
browser_navigate url=<页面URL>   # 重新加载
browser_wait_for time=1
```

**清除 localStorage（模拟新用户）：**
```
browser_evaluate "function=()=>{window.localStorage.clear(); return 'cleared';}"
browser_navigate url=<页面URL>
```

**多用户场景（需要模拟不同身份）：**
- 用 `browser_tabs action=new` 开新标签页
- 同域名 localStorage 共享，清理后刷新即可模拟新用户
- 用 `browser_tabs action=select index=N` 切换标签

### 用例执行模板

```
# TC-001 执行示例
browser_snapshot                              # 确认初始状态
browser_click ref="xxx" element="登录按钮"    # 操作
browser_wait_for text="欢迎"                  # 等待预期结果
browser_snapshot                              # 验证
→ 找到「欢迎」文字 → ✅ 通过
→ 未找到 → ❌ 失败，截图记录
```

## Step 4：输出测试报告

### UI 截图乱码鉴别（重要）

Headless 浏览器在 Linux 容器中**默认没有中文字体**，截图里中文会全部显示为 □ 方块。
这是**截图渲染问题**，不是页面代码缺陷。

鉴别方法：
1. 查看 Playwright snapshot（DOM 文本）—— 如果 snapshot 里文字正常，则页面没问题
2. 用 `fc-list :lang=zh` 检查容器是否有中文字体

报告时需注明：「截图中文显示为方块，为测试容器缺少中文字体所致，非页面缺陷，实际用户设备显示正常。」

### 报告格式

```markdown
## 测试报告

**页面**：<URL>
**执行时间**：<时间>
**用例总数**：X 条（P0: X | P1: X | P2: X | UI: 7）

### 总体结论
✅ 通过 / ❌ 不通过
（P0 全过 = 通过；任意 P0 失败 = 不通过）

### 功能测试结果

| 编号 | 模块 | 描述 | 结果 | 备注 |
|------|------|------|------|------|
| TC-001 | 登录 | 正确账号密码登录 | ✅ | - |
| TC-002 | 登录 | 空密码提交校验 | ❌ | 未出现错误提示 |

### UI 自检结果

| 页面 | 问题描述 | 严重程度 |
|------|------|------|
| P1 登堂入案 | 输入框 placeholder 中文字体显示为方块字符 | P1 |
| 其他页面 | ✅ 无明显异常 | - |

### 失败详情
**TC-002**：点击登录后页面无反应，预期应显示「密码不能为空」提示。

### 建议修复
- TC-002：登录按钮需加表单校验，密码为空时阻止提交并提示用户
- UI-004：placeholder 字体加 system 中文 fallback
```

### 报告交付

- 默认：直接在对话里输出
- 用户要求导出到腾讯文档：
  1. `manage.create_file` 创建 smartsheet，`file_type=smartsheet`
  2. `smartsheet.list_tables` 获取 sheet_id
  3. `smartsheet.update_fields` / `smartsheet.add_fields` 设置列名
  4. `smartsheet.add_records` 写入数据

  **⚠️ 字段值格式（必须用这个，否则数据为空）：**
  ```json
  {"字段标题": [{"text": "内容", "type": "text"}]}
  ```
  Key = 字段标题（不是 field_id），Value = `[{text, type}]` 数组。
  建议通过 Python subprocess 调用，避免 shell 中文转义问题。

## Step 5：AI 自动修复循环

测试报告出来后，如有失败项，**不要等用户说"修复"，直接进入自动修复循环**：

### 修复策略

1. **读取失败用例**：按 P0 → P1 → P2 优先级排序
2. **定位源码**：根据失败描述找到对应的 HTML/JS 代码位置
3. **修复代码**：直接 edit 文件，不要让用户手动改
4. **重新跑测试**：只跑之前失败的用例（回归测试）
5. **循环**：直到所有失败项通过

### 终止条件

- ✅ 所有 P0 + P1 通过 → 进入 Step 6
- ⚠️ 同一用例修复超过 3 次仍失败 → 停止，告知用户无法自动修复，附上失败截图和分析
- ⚠️ 修复导致之前通过的用例失败（回归问题）→ 停止，告知用户

### 修复循环上限

最多循环 **5 轮**，防止无限修复。超出后停止并汇报进度。

## Step 6：通知用户人工介入

所有用例通过后，发送最终通知：

```
🎉 测试全部通过！

通过率：15/15（100%）
修复轮次：X 轮

请你人工介入做最终确认：
1. 打开页面，走一遍主流程，感受实际体验
2. 检查边缘场景（自动化测试覆盖不到的感性部分）
3. 确认 UI 视觉符合预期

文件已更新：<路径>
```

## 注意事项

- **动态内容**：用例预期结果避免依赖具体数值（如「显示3条记录」），改用「显示列表」
- **异步加载**：操作后用 `browser_wait_for text=` 等待结果，不要固定等待时间
- **弹窗处理**：执行前 snapshot 检查有无遮罩/弹窗，先关闭再执行用例
- **登录态**：如页面需要登录，向用户索取测试账号，或请用户先手动登录后再告知我
